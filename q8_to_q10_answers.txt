Question 8:

Best Case O(1):
When the index is 0, the method inserts at the front of the list. It does not need to go through any nodes. It creates
a new node, points its next to the current head, and updates the head pointer. No loops, so this is constant time.

Worst Case O(n):
When the index is equal to the size of the list, the while loop must go through all n existing nodes to find all the 
node before the insertion point. The traversal visits each node once, so this runs in linear time comparable to n.

Average Case O(n):
On average, the insertion position is somewhere in the middle of the list. The expected number of traversals is
about n/2, which is still O(n) since we drop constant factors in asymptotic analysis.

Performance Issues:
One obvious performance issue is that addAt calls this->size() at the start to validate the index. The size() method
itself goes through the entire list from the head node to NULL, which is O(n). This means even a best case insertion 
at index 0 actually costs O(n) because of the size() call at the beginning. The actual traversal to find the
insertion point is also up to O(n), which makes the total worst case O(n) + O(n) = O(n), but with a constant factor 
of about 2x what it needs to be. 


Question 9:

Without changing any of the structure, the most important imporvement would be to maintain a cached size variable in
the List class. Instead of calling size() which goes through the entire list every time (O(n)), we would add an
integer member variable to the List class that starts at 0 and is incremented by 1 in every add method and
decremented by 1 in every remove method.

By doing this, size() would return this->len in O(1) constant time instead of going through the whole list. This 
benefits every method that calls size() or isEmpty() including addAt, removeAt, displayAll, and getAt.


Question 10: 

There are 2 reasons why ~List() was called twice:

1. Near the end of main(), the line "list.~List();" calls the destructor on the list object. This manually triggers cleanup
of all the nodes and increments DN. However, this does NOT deallocate the list object itself because 'list' is a stack
allocated variable and not a heap allocated one.

2. When main() returns and the 'list' variable goes out of scope, C++ automatically calls the destructor for all stack
allocated objects. Since 'list' is declared as "List<int> list;" on the stack, the compiler generates a second call to
~List() as part of normal stack unwinding.
